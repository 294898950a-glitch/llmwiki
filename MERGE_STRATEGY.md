# Merge Strategy

## 目标

本文档定义 `llmwiki` 在 `raw -> wiki` 编译过程中，如何：

- 选择候选页面
- 判断是更新还是新建
- 处理冲突
- 规划后续幂等策略

它是 `LLM_EXTRACTION_DESIGN.md` 的下一层实现约束。

## 核心原则

系统默认立场不变：

- 你是 Wiki 维护者，不是笔记归档器
- 默认优先更新已有知识对象
- 只有在没有可信候选页时，才新建页面
- 不能因为表述变化就轻易重复建页

但系统也不能盲目合并。错误合并的代价高于保守新建，所以要明确候选排序和风险等级。

## 候选排序

候选选择按下面顺序进行。

### 1. Canonical ID 精确命中

如果 raw 已提供 `Canonical ID`，或者 LLM 能稳定提取出一个 `Canonical ID`，则这是最高优先级。

规则：

- 若库内存在完全相同的 `Canonical ID`，默认视为同一知识对象
- 直接进入“更新已有页面”路径
- 除非用户明确要求拆分，否则不新建页面

风险等级：

- 低风险

### 2. 标题精确命中

若没有 `Canonical ID`，则比较 raw 推断出的主题标题与 wiki 标题。

规则：

- 归一化后完全相等，视为强候选
- 若无其他冲突信号，可直接更新

风险等级：

- 中低风险

### 3. Aliases 命中

若标题不完全相等，但候选页的 `Aliases` 中包含 raw 提取到的主实体名或别名，则视为次强候选。

规则：

- 需要结合主题上下文做二次判断
- 不能只因为 alias 命中就盲目合并

风险等级：

- 中风险

### 4. 主题相近但无明确标识

如果只是主题相似、词面接近，但没有 `Canonical ID`、标题精确命中或 alias 强信号，则不能自动合并。

规则：

- 优先停下来让 Claude Code 会话解释候选页差异
- 若仍不确定，保守新建

风险等级：

- 高风险

## 更新还是新建

### 直接更新的条件

满足下面任一条件时，优先更新已有页面：

1. `Canonical ID` 精确命中
2. 标题精确命中，且没有明显冲突
3. `Aliases` 命中，且 LLM 判断是同一知识对象

更新的形式：

- 更新属性：
  - `Verification`
  - `Compounded Level`
  - `Last Compounded At`
  - `Source`
- 追加正文中的 `增量更新` block
- 必要时重写顶部摘要

### 必须新建的条件

满足下面任一条件时，应新建页面：

1. 找不到可信候选页
2. 候选页与 raw 材料属于同名不同实体
3. 候选页只是同一大主题下的邻近概念，但不是同一知识对象
4. LLM 无法给出足够可信的合并解释

### 先停下来问用户的条件

满足下面任一条件时，不应自动落盘：

1. 两个以上候选页都高度相似
2. `Aliases` 命中多个对象
3. 新资料与旧结论明显冲突
4. raw 内容可能需要拆成多个 wiki 页面

## 冲突处理

当前系统不做自动 diff merge，只做 append。

所以冲突处理先采用保守约束：

- 不要静默覆盖旧结论
- 不要因为新资料出现，就直接替换页面旧摘要
- 如果新旧信息冲突，先记录“冲突存在”，再决定是否重写摘要

当前阶段推荐策略：

### 低风险冲突

例如：

- 只是补充数据
- 增加证据
- 增加新的来源

处理方式：

- 直接 append 新的 `增量更新`
- 更新 `Source`
- 更新 `Compounded Level`

### 中风险冲突

例如：

- 同一对象，但结论方向有变化
- 新资料推翻旧资料的一部分判断

处理方式：

- append 新段落
- 在段落中明确写出“新旧判断差异”
- 不自动删除旧 block
- 必要时让 Claude Code 先给出解释，再决定是否重写摘要

### 高风险冲突

例如：

- 可能是不同对象被误并
- 同名但不是同一实体
- 同一 raw 可能应拆成多个页面

处理方式：

- 停止自动写入
- 交给 Claude Code 会话做人工确认

## 当前阶段的代码策略

脚本层当前只适合承担：

- 候选查询
- 属性写回
- 追加增量 block

不适合在脚本层直接决定：

- 哪个候选最合理
- 是否应该覆盖旧摘要
- 是否存在高风险冲突

因此当前阶段：

- `scripts/notion_wiki_compiler.py` 只提供基础候选与写入能力
- 真正的“是否合并”判断，仍由 Claude Code 会话承担

## 幂等性的前置定义

要实现真正的幂等，必须先定义：

- 什么叫“同一个知识对象”
- 什么叫“同一次更新”
- 什么叫“重复执行”

当前暂定：

### 同一个知识对象

满足以下任一条件：

1. `Canonical ID` 相同
2. 标题精确命中
3. `Aliases` 命中且 LLM 判断为同一对象

### 同一次更新

同一个 raw page 多次执行、且 page body 没发生变化，应视为同一次更新。

这意味着未来要补的技术能力包括：

- 记录 raw page 上次编译时的摘要 hash 或 body hash
- 记录该 raw page 最近一次成功写入的 wiki page id
- 若 hash 未变，则避免重复 append

当前还没有这层实现。

## 推荐的下一步实现顺序

1. 在 Claude Code 会话中先固化“候选排序解释模板”
2. 在脚本层增加更细粒度的候选输出
3. 给 raw page 设计可用于幂等判断的字段或状态
4. 再把“重复执行不重复 append”落到代码里

## 当前决定

当前正式决定如下：

- `Canonical ID` 是最高优先级匹配信号
- 标题精确命中是次高优先级
- `Aliases` 只能作为辅助信号，不能单独决定自动合并
- 高风险冲突不自动落盘
- 代码层暂不做自动 diff merge
- 真正的幂等策略要等“同一个对象 / 同一次更新”的定义稳定后再实现
